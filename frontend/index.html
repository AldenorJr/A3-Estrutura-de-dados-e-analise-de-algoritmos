01

Cadastro

Crie sua conta universitária. Se você dirige, marque “Sou motorista” para receber pedidos de carona.

HASH Seus dados são indexados por bairro em uma Tabela Hash de implementação própria — busca em O(1).

02

Cadastrar rota

Informe origem, destino e horário. Se você é motorista, ofereça vagas. Se é passageiro, sua rota vira referência para a busca.

GRAFO Sua rota cria conexões no grafo de afinidade entre usuários — quem mora perto, no horário próximo, vira vizinho.
Faça o cadastro primeiro. Sua sessão será criada automaticamente após salvar o cadastro acima.

03

Buscar carona

Informe o horário desejado. O sistema combina Hash, Grafo e QuickSort para retornar os melhores matches em milissegundos.

QUICKSORT Os matches são ranqueados por um score 0–100: avaliação do motorista, proximidade do bairro, diferença de horário e vagas.

04

Os algoritmos por trás

Três estruturas de dados clássicas trabalhando juntas. Cada uma resolve uma parte distinta do problema.

HASH

Tabela Hash

Por quê? Buscar um aluno por bairro entre milhares seria O(n) numa lista comum. Com hash, é O(1) médio.

busca / insere
O(1) médio
pior caso
O(n)
— usuários indexados
GRAFO

Grafo (lista de adjacência)

Por quê? Compatibilidade não é binária. Bairros vizinhos e horários próximos formam clusters — BFS encontra o componente conectado do solicitante.

adicionar nó
O(1)
BFS
O(V + E)
— bairros conectados
QUICKSORT

QuickSort

Por quê? Os matches precisam vir ordenados. QuickSort divide e conquista — muito mais rápido que ordenação quadrática quando há muitos resultados.

caso médio
O(n log n)
pior caso
O(n²)
— baldes ocupados (hash)
DIJKSTRA

Dijkstra

Por quê? Após o match, calculamos o melhor trajeto motorista → passageiro → UFERSA. Dijkstra encontra o caminho mínimo no grafo ponderado de bairros.

com heap
O((V+E) log V)
sem heap
O(V²)
— rotas no sistema
